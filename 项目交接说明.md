## 1. 项目当前定位

`easy-openclaw` 不是传统可执行程序，而是一个面向 OpenClaw 的“对话式配置向导 Skill”。

它的核心目标是：
- 让用户通过对话完成 OpenClaw 初始化和常见优化
- 尽量减少手动改配置文件和命令行操作
- 按层收集用户选择，再统一写入配置
- 把高频坑位写成硬规则，降低弱模型跑偏概率

当前主流程已经成型，主要是文档规则和参考文件驱动：
- `SKILL.md`：主流程规则
- `references/`：分层说明和执行细则
- `README.md`：面向小白用户的使用说明
- `openclaw插件开发记录.md`：开发过程记录

## 5. 当前最重要的已定位问题

### 5.1 安全审批：目前仍未完成闭环验收

这是当前主线里最重要的未完成项之一。

现状：
- 用户实际测试时，在 `coding` 模式下执行删除文件，没有触发审批。
- 进一步在测试 VM 中执行 `sudo apt update`，同样没有触发审批，而是直接执行成功。

已经确认到的结论：
- 这不等于 Telegram / Discord 审批联动完全失效。
- 真实原因是：OpenClaw 新版里，`tools.profile=coding` 只决定工具集合，不直接等于 `exec` 审批策略。
- 主会话在未沙箱化时，很可能仍是宿主机默认 full access。
- 所以“删文件 / sudo 没弹审批”不能直接作为审批联动失效的证据。

当前仓库已经做过的修正：
- 已在 `SKILL.md`、`layer1-base.md`、`troubleshooting.md` 中明确写入：
  - `coding` 不保证删除文件或 `sudo` 自动触发审批
  - `minimal` 下 exec 往往不可用，审批机制也没有触发机会；若目标是“可执行 + 可审批”，应选择 `coding/full + 审批`
  - 若要测试审批，必须启用 allowlist + `ask=on-miss`，并用“不在 allowlist 的命令”触发审批；同时确保审批提示投递渠道已配置（`approvals.exec mode=session/targets/both`）

但需要强调：
- 这部分目前只是“规则纠偏已完成”
- 还没有完成真正的多渠道审批闭环验收
- 也就是还没有最终确认：Telegram / Discord / Feishu 在正确 exec approval 路径下都能稳定收到审批提示并 `/approve`

因此当前状态应视为：
- 根因已定位
- 文档已修正
- 真实闭环仍待实测

### 5.2 第三轮推荐层跑偏：已修复

这是本次新定位并修掉的一类问题。

现象：
- 第 3 轮有时没有按我们写的推荐清单展示
- 而是错误展示系统自带 Skills，例如：
  - `blogwatcher`
  - `github`
  - `clawhub`
  - `weather`

已经确认的根因：
- 模型在某些轮次会偷懒读取系统 `<available_skills>`
- 或直接枚举本地 `~/.npm-global/lib/node_modules/openclaw/skills/`
- 然后把系统现成的 Skills 误当成我们的推荐清单

当前已做修正：
- `references/layer3-skills.md` 已加硬规则：
  - 默认推荐项只能来自本文件
  - 禁止改用 `<available_skills>`、`openclaw skills list`、`find-skills` 或 skills 目录枚举结果
  - 若输出中出现 `blogwatcher/github/clawhub/weather` 等未列项，必须丢弃重生

这部分建议继续回归测试一次。
